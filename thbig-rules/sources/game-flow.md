## 1. 試合の流れ（時系列）

### 試合前フロー

```
1. judah:「試合ルール」        → sian が球場・DH制・日付・季節・開始時間を表示
2. 両者 1d20                    → 出目の大きい方が後攻（judah=14 > mori=11 → judah後攻）
3. 先攻(mori)「予告先発：26 綿月豊姫」宣言
4. 後攻(judah)「先発投手 97 LL 6/1 サリエル…」宣言
5. judah:「天候チェック」       → sian が天候（晴天/雨天）を表示
6. 先攻(mori) オーダー発表：スタメン・控え・投手・登板履歴をチャットに貼り付け
7. 後攻(judah) オーダー発表：同上
8. 先攻(mori) 1d20 怪我チェック → 2以下で誰かが怪我（10 → 怪我なし）
9. 後攻(judah) 1d20 怪我チェック → 5 → 怪我なし
10.「試合開始」宣言（先攻投手・打順発表）
```

### 1イニングの繰り返し

```
[表（先攻打撃）]
  守備側(judah):「{投手背番号}-p」
  ↓
  sian: 投球ナンバーを出力（例: result:３＊）
  ↓
  攻撃側(mori):「{打者背番号}-{投球番号}」
  ↓
  sian: 打撃結果を出力（例: result:Ｇ６Ｄ）
  ↓
  必要に応じて追加判定（レンジ・エラー・UP・盗塁等）
  ↓
  攻撃側がテキストで状況を宣言（例: 「遊ゴ　１死」）
  ↓
  3アウトになるまで繰り返し

[裏（後攻打撃）]
  同様（守備=mori、攻撃=judah）
```

## 2. ボットコマンド（sian）の体系

### 2-1. 投球コマンド

**書式**: `{背番号}-p`

```
入力(judah): 97-p
出力(sian): (judah) サリエル Dice => [8] result:３＊
```

- 守備側（投手を持つチーム）が打つ
- 出力の `result:` に投球ナンバー（1〜5）が出る
- `＊`付き（例: `３＊`）は疲労状態の投手。疲労劣化ルール適用
- 特殊表記:
  - `/` → 左右投手で投球ナンバーが変わる（例: `３／４` = 左投手→3、右投手→4）
  - `NPN` → 走者有無で変わる（例: `３Ｐ１` = 走者なし→3、走者あり→1）

**実例（特殊投球ナンバー）**:

| ログ出力 | 意味 | 使う番号の決め方 |
|---------|------|----------------|
| `３＊` | 投球3（疲労状態） | 1段階劣化して「2」として扱う |
| `３／４` | 左/右で分岐 | 左投手ならスラッシュ左側（3） |
| `３Ｐ１` | 走者なし/あり | 走者なし→3、走者あり→1 |

### 2-2. 打撃コマンド

**書式**: `{背番号}-{投球番号}`

```
入力(mori): 39-3
出力(sian): (mori) 咲夜 Dice => [10] result:Ｇ６Ｄ
```

- 攻撃側（打者を持つチーム）が打つ
- 投球コマンドの `result:` に出た番号を使う
- `/` 記号の場合: 左投手が相手なら `/` の**左側**を使う

**打撃結果の種類（ログで確認）**:

| `３Ｈ９` | 三塁打（9=ライト方向） | 「右３　無死３塁」 |
| `２Ｈ７ａ` | 走者3進塁の二塁打 | 「左２　１死２塁」 |
| `Ｈ９ａ` | 走者2進塁の単打 | 「右安①　2-0」 |
| `Ｇ６Ｄ` | 遊撃方向ダブルプレイ | 「遊ゴ　チェンジ」 |

### 2-3. UP表（好プレイ珍プレイ表）

**書式**: `打:up`

```
入力(mori): 打:up
出力(sian): (mori) Dice => [1,17] result:投手エラーチェック
```

- 打撃結果が `ＵＰ` になった際に使う
- d20を2個振って4桁の番号から特殊イベントを決定
- 結果に応じてさらに追加コマンドが必要になる

**ログで確認したUPイベント**:

| UPロール | イベント | 次の処理 |
|---------|---------|--------|
| [1, 17] | 投手エラーチェック | 守備側: `内野エラー{ランク}` |
| [2, 16] | 捕手エラーチェック | 守備側: `内野エラー{ランク}` |
| [19, 19] | 強烈な投ゴロ Ｇ１ａ 投手が負傷退場 けが表参照のこと | 攻撃側: `怪我レベル{N}` |

### 2-4. レンジチェック

#### 内野レンジチェック

**書式**: `内野レンジ{守備範囲数値}`

```
入力(judah): 内野レンジ４
出力(sian): (judah) Dice => [6] result:ＩＨ
```

**結果の種類**（ログで確認）:

| result | 意味 |
|--------|------|
| `Ｈ` | レンジオーバー → ヒット扱い |
| `ＩＨ` | 内野安打 |
| `ＧＤ` | ダブルプレイ |

#### 外野レンジチェック

**書式**: `外野レンジ{守備範囲数値}`

```
入力(mori): 外野レンジ４
出力(sian): (mori) Dice => [5] result:Ｈ
```

**結果の種類**（ログで確認）:

| result | 意味 |
|--------|------|
| `Ｈ` | レンジオーバー → ヒット扱い |
| `Ｆａ` | 外野フライ（通常アウト） |

#### 投手レンジチェック

打撃結果が `Ｐ`（投手方向）のとき、コマンド上は `内野レンジ{投手の守備範囲}` で判定する。

```
ログ例:
(judah) 投手レンジ０    ← 情報として宣言
内野レンジ０            ← 実際のコマンド
(judah) Dice => [4] result:Ｈ
```

### 2-5. エラーチェック

**書式**: `内野エラー{エラーランク}` または `外野エラー{ランク}`

```
入力(judah): 内野エラーC
出力(sian): (judah) Dice => [13] result:Ｇｆ
```

**結果の種類**（ログで確認）:

| エラーランク | result | 意味 |
|------------|--------|------|
| A | Gf | 問題なし→通常内野ゴロ処理 |
| C | Gf | 問題なし→通常内野ゴロ処理 |

※ランクSはエラーチェック不要（確定アウト）

### 2-6. 盗塁コマンド

#### スタートチェック

**書式**: `{背番号}-st{塁}`

```
入力(mori): 45-st1
出力(sian): (mori) 夢美 【st値：７ ed値：１５】 Dice => [5] result:ランナースタート！
```

- 出力に st値・ed値が表示される
- `ランナースタート！` = スタート成功
- 左投手に対してスタート値が足りない場合: `※ただし左投手ならスタート切れず` の注記が付く

#### 成功判定

**書式**: `{背番号}-ed{塁}`

```
入力(mori): 45-ed1
出力(sian): (mori) 夢美 【st値：７ ed値：１５】 Dice => [3] result:盗塁成功！
```

**盗塁の全流れ**（ルール: 盗塁は即座に解決 = 投球前に完結）:

```
攻撃側: 盗塁を宣言（チャットに「盗塁」と書く）
守備側: 同意（「どうぞ」等）
攻撃側: {背番号}-st{塁} → スタート判定
  → 成功なら: {背番号}-ed{塁} → 成功判定
  → 失敗なら: 盗塁失敗（盗塁不可状態になる）
その後: 通常の打席へ
```

**左投手への盗塁制限の実例**（2回裏）:

```
judah: 純狐盗塁
mori: どうぞ
judah: 110-st1
sian: 純狐 【st値：６ ed値：１５】 Dice => [5] result:ランナースタート！ ※ただし左投手ならスタート切れず
judah: H  ← ヒッティングに変更（盗塁中止）
→ 次の打席へ
```

→ 豊姫（左投手）に対して st値：6、サイ目：5。左投手修正-2で実効4未満 → スタート切れず。judahは盗塁を断念してヒッティングに変更。

### 2-7. タッチアップ判定

**書式**: `b-{外野手の方向番号}`

```
入力(mori): b-4
出力(sian): 通常:12(6)以下 タッチアップを除く2死時:14(8)以下 F8aの2走:14以下 F9aの2走:18以下 セーフ
```

- コマンド単体では閾値表を表示するだけ
- その後プレイヤーが自分で `1d20` を振って判定する

```
mori: b-4        → sian が閾値表示
mori: 1d20       → [20] → 20 > 12 なのでアウト
mori: 左飛　本塁タッチアウト
```

### 2-8. 怪我判定

**書式**: `怪我レベル{N}（1〜6）`

```
入力(mori): 怪我レベル５
出力(sian): (mori) Dice => [12] result:怪我3日
```

- UPイベントや試合前怪我チェックで使う
- 出力: `怪我X日` または `問題なし` 等

### 2-9. セットアップ系コマンド

| コマンド | 使用者 | 出力例 |
|---------|--------|-------|
| `試合ルール` | 後攻（試合前） | 球場・DH制・日付・季節・開始時間 |
| `天候チェック` | 後攻（試合前） | Dice [8,20] result: 晴天 |
| `1d20` | 両者（先攻後攻・怪我チェック等） | judah -> 1D20 = [14] = 14 |

## 3. ゲームルールとログの対応

### 打撃結果の適用

| ルールブックの記述 | ログでの現れ方 |
|-----------------|--------------|
| 守備側がd20→投球ナンバー | 守備側が `{背番号}-p` → sian が投球ナンバーを出力 |
| 攻撃側がd20→打撃結果 | 攻撃側が `{背番号}-{投球番号}` → sian が結果を出力 |
| レンジチェック（守備側コマンド） | 守備側が `内野レンジN` / `外野レンジN` |
| エラーチェック | UP後に守備側が `内野エラー{ランク}` |
| UP表 | 攻撃側が `打:up` |

### 投手疲労（＊マーク）

- `３＊` = 疲労状態の投球3 → 実際は「2」として処理（1段階劣化）
- こいし（51）は4回裏から `*` 付きで登板: `51-p *` という形で入力

```
ログ: (mori) 51-p *
出力: (mori) こいし Dice => [13] result:３Ｐ１
```

→ `51-p *` の `*` は付加情報（ログ表示用）で、sianのコマンドとしては `51-p` でも同じ動作をするものと思われる

### 特殊投球ナンバーの判断フロー

```
投球ナンバー出力が通常数字（1〜5）
  → そのまま打撃コマンドに使う

投球ナンバー出力が「N／M」（スラッシュ）
  → 左投手 → 左側（N）を使用
  → 右投手 → 右側（M）を使用
  → 判断: 守備側が「L」または「R」と発言して確認することがある

投球ナンバー出力が「NPN」
  → 走者なし → 左側（N）
  → 走者あり → 右側（M）
  → 判断: 攻撃側が状況を確認して打撃コマンドを送る
```

### 前進守備

- 宣言: 守備側が「内野前進」とチャットに書く（打者ごとに宣言必須）
- 適用: 打撃結果表の「前進守備」欄を使う
- 実例（5回表）:
  ```
  大妖精: 3H8（無死3塁）
  judah: 内野前進
  くるみ: G4f（二ゴ・フォースアウト） → 1死3塁（3塁走者は動けず）
  ```

### 投手交代

```
judah:「投手交代：　サリエル　→　ミスティア」
judah: 新投手のスペック貼り付け（登板数・投球回・防御率等）
mori:「交代了解です」
→ 次の打席から新投手のコマンドを使う
```

### 代打・守備交代

```
mori:「大妖精→代打：相良」
judah:「代打了解です」
→ 打撃コマンドで代打選手の背番号を使う（F95-5）

守備交代例:
mori:「守備：相良→二　センターにいた妖夢→右：ラルバ　右：はたて→中」
judah:「交代了解です」
```

## 4. 気づいた点・疑問点

### ✅ 確認できた事項

1. **盗塁は投球前に完結する**: ルールブック通り「バント・盗塁は即座に解決」が確認できた。`st1` → `ed1` の2コマンドで完結し、その後次の打席が始まる。

2. **左投手と盗塁制限の実際の処理**: `st1` の出力に `※ただし左投手ならスタート切れず` という注記が付く場合がある。これは sian が左投手フラグを検知して出力するものと思われる。プレイヤーは注記を見て盗塁を断念するかどうかを判断し、チャットで宣言する。

3. **タッチアップは2段階**: `b-{N}` で閾値表を出力させ、プレイヤーが別途 `1d20` を振る。コマンド自体はダイスを振らない。

4. **UPイベントは多様**: `打:up` で2d20を振り、出た番号で「投手エラーチェック」「捕手エラーチェック」「負傷退場」等が発生。エラーチェックへの連鎖がある。

5. **先発投手スペックはチャットに貼る形式**: `先発投手 97 LL 6/1 サリエル 7登板 42 1/3回 4.04 2勝 2負 57% 30三振 全快` のように自由記述で貼り付ける（ボットへのコマンドではなく情報共有）。

6. **得点は攻撃側がテキストで宣言**: `右安① 2-0` → `①` が得点数（丸付き数字）、`2-0` が現スコア（攻撃チーム-守備チーム）。

7. **`51-p *` の `*` 記法**: 疲労状態の投手を投げさせる際に `*` を付加しているが、これは情報表示用と思われる（sian のコマンド構文的には不要の可能性がある）。

8. **攻撃側がスコアを宣言する**: `「右安① 2-0 1死満塁」` のように攻撃側がスコアと状況を合わせて宣言することが多い。

### ❓ 疑問・不明点

1. **`51-p *` の `*` は実際にコマンドとして意味を持つか？** `sian` への投球コマンドでは `*` がオプションになるのか、それとも単なるコメント（スペース前の注記扱い）なのか不明。実装時に要確認。

2. **レンジチェック結果の詳細**: `内野レンジ0` → `result:Ｈ` は「ヒット」だが、`内野レンジ0` → `result:ＧＤ` もあった（7回裏ナズーリン打席）。同じ範囲0なのに結果が異なる。これはダイスによる確率判定であり、範囲値が閾値として機能していると推測されるが、実際の判定テーブルの確認が必要。

3. **`打:up` の `打:` プレフィックス**: 攻撃側が打つのに `打:up` となっている。他の `up` 系コマンド（特徴発動等）と区別するためのプレフィックスか？

4. **投球ナンバーの `L` / `S` 宣言**: 6回表のログで:
   ```
   89-3 → result:Ｈ９/Ｇ３ｆ
   judah: L
   mori: 右安
   ```
   `L` = 左投手なのでスラッシュ左側（H9）を使う、という意味と解釈。しかし直後:
   ```
   34-3 → result:Ｈ９/Ｋ
   judah: L
   judah: S
   mori: 左安
   ```
   `S` の意味が不明。走者ありで別の条件が絡むケース？要調査。

5. **先攻後攻と怪我チェックの順序**: ログでは先攻（mori）が先に怪我チェックを行い、後攻（judah）が後に行っている。これが必ずこの順序かは1サンプルでは判断できない。

6. **背番号の管理方法**: judah の 023（青木宣親）は、sian の出力で「mori」と表示されている（`(judah) mori Dice => [11] result:Ｋ`）。これはアカウント名と選手名の関係の問題と思われる。IRC bot がどのようにキャラクター登録を管理しているかは要確認。

7. **「H」の宣言の使い方**: ログ中に judah が単独で `H` と発言する場面がある。文脈から「ヒッティング（作戦変更なし）」または「ヒット（確認）」を意味するようだが、正式な意味は不明。

## 6. IRC実装への示唆

1. **sian のコマンド体系は確立している**: ユーザーは決まったコマンドを打つだけで自動判定される。実装では同じコマンドをサポートする必要がある。

2. **状況宣言はプレイヤー責任**: 走者状況・アウトカウント・スコアはプレイヤーが自分でテキスト宣言する。Webクライアントでこれを自動化・可視化できると誤記が減る。

3. **投球コマンドと打撃コマンドが交互に来る**: 守備側と攻撃側が交互にコマンドを送る構造。これをターン管理として実装できる。

4. **複数のサブコマンドが連鎖する**: UP → エラーチェック、外野フライ → タッチアップ判定のように、1打席で複数のコマンドが必要になるケースがある。状態管理が重要。

5. **フォーマットは半自由**: スタメン発表や試合状況の宣言はプレイヤーの自由記述。コマンドのみ統一フォーマット。Webクライアントは構造化フォームでサポートすると良い。
