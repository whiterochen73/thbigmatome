# 第4章: 試合前準備

**状態**: 実装済み（cmd_915 / subtask_915b）
**品質基準**: エミュレーターが本章の処理を実装できるレベル

---

## 概要

この章は **Phase3_GameSetup**（試合前準備フェーズ）を扱う。
試合開始宣言（Phase4_InGame 遷移）までに必要なすべての手順——ホーム/ビジター決定から予告先発・天候チェック・オーダー発表・負傷チェックまで——を定義する。

```
Phase2_Season → [対戦カード確定・試合日到達] → Phase3_GameSetup
Phase3_GameSetup → [天候=雨天中止] → Phase3_GameSetup（順延ループ）
Phase3_GameSetup → [全手順完了] → Phase4_InGame
```

関連章:
- § 第5章: 打席処理（試合開始後のメインループ）
- § 第9章: 負傷判定（試合前負傷チェックのプロトコル詳細）

## 4.1 試合準備フロー（完全手順）

```
GAME_SETUP:
  INPUT: season_schedule（対戦カード・球場・日付）

  STEP 1: 試合ルールマクロ実行
    → sianボットが以下を出力:
      球場名・DH制有無・日付・季節・開始時間
    → 球場IDから球場属性を取得（§4.3参照）

  STEP 2: ホーム/ビジター決定
    両者が1d20を実行
    大きい方がホーム（後攻）
    同値 → やり直し（再度1d20）

  STEP 3: 予告先発
    両者が先発予定投手を発表
    オープナー宣言もこのタイミングで行う
    オープナーの場合 → 第二先発（続投可能投手）も同時に宣言

  STEP 4: 天候チェック（§4.4参照）
    雨天中止の場合 → 順延処理 → STEP 3に戻る（ループ）
    催行可能 → STEP 5へ

  STEP 5: オーダー発表（ビジター先 → ホーム後）
    各チームが以下を発表（§4.2参照）:
      (a) 1,2軍入れ替え発表（変更がある場合のみ）
      (b) メンバー発表

  STEP 6: 試合前負傷チェック（→ 第9章 §9.3）
    各チームが1d20 ≤ 2 → 負傷判定プロトコル発動
    負傷発生時 → オーダー変更・出場選手登録変更が可能

  STEP 7: 試合開始宣言
    両チームの準備完了後、挨拶して試合開始
    → Phase4_InGame に遷移

  OUTPUT: GameState（初期化済み。§1.1参照）
```

## 4.2 オーダー発表（メンバー発表の必須記載事項）

```
MEMBER_ANNOUNCEMENT:
  必須記載:
    - チーム名
    - 日程進捗（チーム日程表の試合番号）
    - 特例指定選手（いる場合）
    - スターティングメンバー（打順番号 + 守備位置）
    - ベンチ入り選手
    - ベンチ外選手（「オフ」= 戦略的ベンチ外）
    - 投手の疲労状況（「全快」= 累積0イニング）
    - 登板履歴（最低7日分）
    - （中継）マーク: リリーフ契約の投手は記載義務あり
    - 借金特例（ある場合のみ）

  書式の個性:
    記載書式はプレイヤーごとに個性が大きい。
    パーサーは固定フォーマットを前提にしないこと。
```

## 4.3 DH制

```
DH_RULES:
  基本: NPB準拠

  DH制なしの場合:
    投手が打順に入る（9人打順）

  DH制ありの場合:
    投手は打順に入らない（DHが投手の代わりに打順に入る）

  大谷ルール（リアル大谷ルール）:
    発動条件: スタメン発表時に「投手」と「DH」が同一選手
    効果: その投手がマウンドを降りた後もDHとして打席に残留できる
    分類: 「交代」ではないためリエントリー制限に抵触しない

  投手DH降板後の扱い:
    投手として降板してもDHの地位は失わない
    打順はそのまま引き継ぐ
```

## 4.4 球場設定

```
STADIUM:
  各球場は以下の属性を持つ（球場ごとに個別に設定）:

  属性一覧:
    - ドーム / 屋外（→ 天候判定に影響 §4.5）
    - 人工芝 / 天然芝（→ 天候判定に影響 §4.5）
    - 球場の広さ（→ UP表1911-1914の結果に影響）

  球場と UP表の関係:
    UP表1901-1910: 捕手レンジ（球場共通）
    UP表1911-1914: 球場固有イベント（球場ごとに結果が個別に固定）

    球場の広さと UP表1911-1914 の対応:
      狭い球場 → 大飛球 → HR
      広い球場 → 大飛球 → フライアウト

  注意: 「球場タイプ1112/1314」という固定分類は存在しない。
        UP表の種類は球場ごとの個別設定によって決まる。
```

## 4.5 天候チェック

```
WEATHER_CHECK:
  実行タイミング: 予告先発（STEP 3）の後

  中止確率の目安（球場属性による）:
    ドーム球場: ほぼ中止なし（1/200 程度）
    人工芝（屋外）: 低確率
    天然芝（屋外）: 人工芝よりわずかに高い
    雨期: 中止確率が上がり約10%

  ※ 具体的なダイス値はルールブック未記載（マクロ内部実装に委ねる）

RAIN_POSTPONEMENT:
  中止時の処理フロー:
    1. 試合を順延
    2. 予備日を使用（交流戦期間は「交流戦予備日」を優先）
    3. 予告先発の再宣言（変更なしなら「スライド」宣言で省略可）
    4. 天候チェックを再実行
    5. 催行可能になるまで繰り返す（ループ）

  予備日使用の判断:
    通常: プレイヤー判断
    9月以降 OR 予備日残3日以下 → 運営（コミッショナー）判断に切替

RAIN_DURING_GAME（試合中の雨天処理, §1.3 参照）:
  屋外球場 + 4回終了以前 → ノーゲーム（成績無効、投手疲労は有効）
  屋外球場 + 5回終了以降 → 雨天コールド（試合成立）
  ドーム球場 → 何も起こらず、投球やり直し
  ドーム球場 + 試合成立済み → 即雨天コールド（ドームでも成立後は例外）
```

## 4.7 試合成立条件

```
GAME_VALIDITY:
  ゲームが「試合成立」として記録されるための条件:
    屋外球場での降雨コールド: 5回終了以降
    延長戦上限: 12回（§10.2参照）

  ノーゲーム（成績無効）の条件:
    屋外球場での降雨: 4回終了以前
    ※ ノーゲームでも投手疲労の変化は有効（消耗した疲労は戻らない）
```
