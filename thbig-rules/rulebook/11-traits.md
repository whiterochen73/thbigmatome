# 第11章: 特徴

**状態**: 実装済み
**品質基準**: エミュレーターが特徴の発動判定・効果適用・消化順序を実装できるレベル

---

## 11.1 概要

「特徴」は選手カードに記載される固有の能力であり、試合中に特定条件で発動して通常のルールに修飾を加える。

公式には投手特徴・野手特徴・特殊能力を含めてすべて「特徴」と呼ぶ。本章では実装上の便宜のため、発動メカニズムに基づいて4つの区分に分類する。

---

## 11.2 分類体系

### 4区分の定義

| 区分 | 発動タイミング | 説明 | 例 |
|------|-------------|------|-----|
| トリガー型特徴 | イベント駆動 | UP表の「特徴確認」出目や負傷チェックのヒット時に発動 | 1C5, けが○日, DB, WP |
| パッシブ型特殊能力 | 常時適用 | 条件判定なしで常に効果が適用される | WPしない, 風神少女 |
| バイオリズム | 期間限定 | 二十四節気ベースの期間中、他の特徴を差し替える修飾子 | （後述） |
| 固有（個別） | 複合条件 | 上記パターンに収まらない複合的な特徴 | 渡里ニナの「けが100日」 |

### 区分の判定フロー

```
特徴の文字列を読み取り:
  ├─ 常時効果キーワード（「しない」「クイック」「風神少女」「精神疲労」）
  │   → パッシブ型特殊能力
  ├─ 二十四節気の日付範囲を含む
  │   → バイオリズム
  ├─ data/traits.yaml の汎用パターンにマッチ
  │   → トリガー型特徴
  └─ いずれにもマッチしない
      → 固有（個別）特徴として選手データに記述
```

---

## 11.3 発動タイミングと消化順序

### トリガー型の発動タイミング

| トリガー | 発動する特徴 |
|---------|------------|
| UP表で「投手特徴確認」を引いた | 投手特徴 |
| UP表で「打者特徴確認」を引いた | 野手特徴 |
| 負傷チェックがヒットした | けが特徴（＊ルール） |

### 特徴不在時のデフォルト

| 状況 | デフォルト結果 |
|------|-------------|
| 投手特徴を持たない野手が投手として登板中に「投手特徴確認」を引いた | **ホームラン**（被弾） |
| 野手特徴を持たない投手が打席に立って「打者特徴確認」を引いた | **三振** |

### けが特徴の消化順序（＊ルール）

カードの特徴欄にけが免除・日数指定が記載される場合、上から順に消化する。

```
処理順:
  ＊（免除マーク）      → 怪我せず判定終了。1回使い切り
  日数のみ（例: 3日）    → その日数分負傷
  日数＋（するごと）     → 毎回その日数の怪我（消化されず繰り返し）
  （するごと）のみ       → けがレベルで怪我チェック（特徴けが扱い）
  全エントリ消化後       → カード記載のけが特徴レベルで通常判定に戻る
```

→ 負傷判定の詳細は第10章を参照

## 11.4 投手特徴（トリガー型）

UP表で「投手特徴確認」を引くと発動する。

| type | 特徴名 | 効果 |
|------|--------|------|
| wp | WP | 暴投。走者1進塁して再投球。走者なし時は投球から振り直し |
| db | DB | 死球（怪我・危険球退場は発生しない） |
| balk | ボーク | 走者1進塁して再投球。走者なし時は投球から振り直し |
| home_run | ホームラン | ホームランを被弾 |
| injury | けが | 投手が自分のけがレベルで怪我チェック |
| injury_days | けが＊＊日 | その日数分怪我して降板。投手交代が必要 |
| vs_right | 対右 | 右打者なら投球5、左打者なら投球1で打つ |
| vs_left | 対左 | 右打者なら投球1、左打者なら投球5で打つ |

---

## 11.5 投手特殊能力（パッシブ型）

条件判定なしで常に適用される投手の能力。

| type | 能力名 | 効果 |
|------|--------|------|
| no_wp | WPしない | あらゆる暴投を防ぐ。UP表の暴投は投球から振り直し。パスボールは防げない |
| no_balk | ボークしない | ボークを防ぐ。UP表のボークは投球から振り直し |
| mental_fatigue | 精神疲労 | 通常の疲労に加え、失点時点で疲労状態になる |
| quick_positive | クイック○ | 盗塁されにくい。盗塁st値出目+2、捕手肩+2 |
| quick_negative | クイック✕ | 盗塁されやすい。盗塁st値出目−2、捕手肩−2 |

### 専属捕手

特定の捕手とペアで投球が有利になる仕組み。

- カードの特徴欄に捕手名が記載される
- 専属捕手とバッテリーを組むと、専用の投球スタイルが適用される
- 専属捕手なしでの起用は可能だが、投手と捕手の両方がペナルティ対象
- 他の投手の登板時に専属捕手を起用した場合は、捕手のみがペナルティ対象

### 走者有無分岐（投球スタイル）

投手が走者の有無によって投球スタイルを切り替えるパターン。カード上は「(有走者)」「(無走者)」等の条件付きで記載される。

## 11.6 野手特徴（トリガー型）

UP表で「打者特徴確認」を引くと発動する。

| type | 特徴名 | 効果 |
|------|--------|------|
| db | DB | 死球（怪我・危険球退場は発生しない） |
| db_injury | DBけが＊＊日 | 死球で日数分怪我。代走が必要 |
| db_brawl | DB乱闘 | 死球で乱闘、打者退場。投手はけがレベル6で怪我チェック |
| injury_days | けが＊＊日 | 日数分負傷。代打が必要 |
| clutch_1c5 | 1C5 | 走者なし時は投球1で振り直し、走者あり時は投球5で振り直し |
| clutch_chance | チャンス | 走者なし時は投球5で振り直し、走者あり時は投球1で振り直し |
| infield_hit | 内野安打 | 内野安打。1塁走者ありなら2塁フォースアウト。前進守備時は3塁走者がホームアウト |
| vs_right | 対右 | 右投手なら投球1で振り直し、左投手なら投球5で振り直し |
| vs_left | 対左 | 右投手なら投球5で振り直し、左投手なら投球1で振り直し |
| pinch_hit | （代打） | 代打時は投球1で振り直し。代打でない時は他の特徴を使用 |

## 11.8 バイオリズム（期間修飾子）

### 概要

バイオリズムは二十四節気ベースの期間限定修飾子。該当期間中、選手の特定の特徴が別の特徴に差し替えられる。

パッシブ型に近いが、効果が「特徴の書き換え」であるという点で独立した区分とする。

### 仕組み

```
バイオリズム定義:
  period: 二十四節気の期間（例: 「立春〜雨水」）
  replaces: 差し替え対象の特徴
  replacement: 差し替え後の特徴
```

- 該当期間中は `replaces` で指定された特徴が `replacement` に置き換わる
- 期間外は通常の特徴が適用される
- カード上では日付範囲 + 効果 + 別条件の効果として複数行にわたって記載される

### 二十四節気

試合日の日付で判定する。IRC上で「試合ルール」マクロを実行した際に表示される季節情報と連動する。

### データ上の扱い

- カードからの読み取り時は複数文字列に分割されて `traits` 配列に格納される（3分割が多い）
- 将来的に専用フィールドへの昇格を検討

---

## 11.9 固有特徴（パターン外の扱い方針）

### 方針

上記の汎用パターン（トリガー型・パッシブ型・バイオリズム）に収まらない複合条件の特徴は「固有特徴」とする。

```
原則:
  - 汎用パターンに収まる → data/traits.yaml に type + パラメータで定義
  - パターン外             → 選手データとして個別に記述
```

### 具体的な処理方法

固有特徴の具体的な処理方法は、Clubhouse（監督向けシステム）の設計時に決定する。現時点では以下の枠のみを用意する。

| 選手例 | 特徴内容 | 備考 |
|--------|---------|------|
| 渡里ニナ | けが100日 | けがレベル不要。特徴で日数固定 |
| 鈴仙（今季絶望） | 今季絶望 | けがレベル不要。特徴で固定 |

固有特徴を持つ選手の一覧と処理ルールは、選手カードデータの側で管理する（本章のスコープ外）。

---

## 11.10 data/traits.yaml 仕様

汎用パターンの特徴を type + パラメータで定義するYAML。

### 設計原則

1. **汎用パターンのみ**: 個別特徴はここに入れない（選手データ側で持つ）
2. **拡張可能**: 新しい特徴 type を追加するだけで拡張できる構造
3. **メカニクスとデータの分離**: 本章（11-traits.md）が仕組みを定義し、data/traits.yaml がパターンデータを持つ

### ファイル構造

```yaml
# data/traits.yaml
version: "1.0"
traits:
  - type: "type_name"
    category: "pitcher_trigger | batter_trigger | pitcher_passive | batter_passive"
    name_ja: "日本語表示名"
    effect: "効果の説明"
    parameters:
      key: value
```

→ 詳細は `rulebook/data/traits.yaml` を参照
